# Модель данных: Удаление задачи (004-task-delete)

**Фича**: 004-task-delete  
**Дата**: 2026-02-16  
**Основа**: существующая модель Task (001, 003); новых полей не вводится.

---

## Операция удаления

Удаление задачи — единственное изменение в модели данных: задача исключается из массива и из персистенции.

| Действие | Вход | Поведение |
|----------|------|-----------|
| deleteTask(id) | id: string | Найти задачу по id; при наличии удалить из массива `tasks`, вызвать сохранение в localStorage. При отсутствии задачи с таким id — no-op (ничего не делать). |

**Правила**:
- Удаление безвозвратное: восстановление в рамках фичи не предусмотрено.
- После удаления массив пересохраняется в тот же ключ `todo-app-tasks`; порядок оставшихся задач не меняется (индексы/order не пересчитываются для остальных, если не требуется иное — см. ниже).
- При удалении задачи из группы приоритета остальные задачи в этой группе сохраняют свои `order`; при необходимости консистентности порядка (например, сжатие 0,1,2,...) это можно делать в реализации по соглашению проекта.

**Персистенция**:
- Тот же ключ и формат, что в 001/003: `localStorage['todo-app-tasks']` = JSON-массив объектов Task.
- После `deleteTask(id)` массив без удалённой задачи сериализуется и записывается в localStorage.

---

## Сущности (без изменений)

- **Task**: id, text, completed, order, priority — как в 003-task-priority. Удалённая задача просто отсутствует в массиве.

---

## Валидация и граничные случаи

- Вызов `deleteTask(id)` с несуществующим id: не менять массив, не сохранять (no-op).
- Пустой список после удаления последней задачи: UI показывает пустое состояние (как при отсутствии задач в 001).
